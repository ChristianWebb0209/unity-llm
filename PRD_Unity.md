# PRD: Unity Port of `godot-llm` (FastAPI RAG + XML Tool-Calling Fine-Tune)

## 0. Overview
This project ports the existing `godot-llm` architecture to Unity:
- FastAPI Python backend that builds prompt/context and runs a tool-call loop.
- No backend user accounts and no backend user database.
- Unity Editor plugin is the product: it gathers editor context, calls the backend, and executes returned tool actions locally.
- Fine-tune a model to emit XML-wrapped tool calls for Unity editor tooling.

The Unity workspace generated by `setup_unity_prd_workspace.ps1` contains reusable copies of:
- `rag_service/` backend code (FastAPI endpoints, prompts, tool registry).
- `fine_tuning/` Composer v3 contract enforcement scripts + Colab training script.

## 1. Goals
1. Editor-native "AI assistant" workflow inside Unity:
   - Fix compile/CS errors for a script you are editing.
   - Create/modify scripts and GameObjects.
   - Wire events (e.g. Button.onClick -> method).
   - Apply safe, bounded iterations: apply tools -> compile/test -> follow-up with errors.
2. Tool-calling reliability:
   - Strict tool schema contract.
   - Strict XML wrapper contract: `<tool_call>{ "name": "...", "arguments": {...} }</tool_call>`.
   - Unknown tools and malformed tool-call outputs must fail hard in dataset validation and release gates.
3. Fine-tuning pipeline:
   - Build sample training data using OpenAI calls (AGENT and ASK modes).
   - Enforce strict dataset normalization (Composer v3 rebuild scripts).
   - Train on Colab (LoRA/QLoRA).
   - Run post-train inference contract suite.

## 2. Non-goals (first release)
1. No authentication or per-user persistence on the backend.
2. No vector database for runtime retrieval of user projects.
3. No hosted chat history storage; all chat state lives in the Unity plugin.
4. No "auto-prompting across the whole project" without tool/diff context.

## 3. System Architecture

### 3.1 Components
1. Unity Editor plugin (main UX + tool executor)
   - Tabs: Chat, Pending & Timeline (diff preview), Settings.
   - Streaming UI: typewriter-like incremental rendering of backend answer.
   - Tool execution: deterministic mapping from backend tool calls to Unity editor operations.
   - Local state only: chat transcripts, pending edits, undo/revert state.

2. FastAPI backend (stateless w.r.t. user data)
   - Endpoints:
     - `GET /health`
     - `POST /query` (answer + snippets)
     - `POST /query_stream_with_tools` (stream answer + tool-call suffix)
     - `POST /composer/query` (fine-tuned endpoint, tool_calls emitted directly)
     - `POST /composer/query_stream_with_tools` (stream variant)
   - Contract:
     - Tool schema is the source of truth (exported from backend).
     - Tool loop is bounded by `max_tool_rounds`.
   - Environment:
     - Load root `.env` on startup.
     - No server persistence of user projects/chats.

3. Fine-tuning pipeline (Composer v3)
   - Generate dataset records that match the strict XML tool-call contract.
   - Validate datasets against:
    - `tools.json` (repo root)
     - `fine_tuning/schemas/composer_v3_tool_contract.json`
     - `fine_tuning/schemas/composer_v3_tool_aliases.json`
   - Train and release-gate:
     - pre-train gates
     - post-train inference contract suite

## 4. Environment Requirements (.env + venv forwarding)

In the Unity workspace:
- Root has a single `.env` file.
- `rag_service/.venv` and `fine_tuning/.venv` are created locally by you.
- All scripts must load the root `.env` (shared OpenAI keys and model configuration).

Operationally:
1. Create venvs:
   - `cd rag_service; python -m venv .venv`
   - `cd fine_tuning; python -m venv .venv`
2. Ensure the backend loads root `.env` (backend code already uses relative root loading).
3. Ensure fine-tuning scripts locate the repo root (they rely on finding `rag_service/` or fixed parents depth).

## 5. Unity Tool Schema: What you must implement

### 5.1 Tool categories
You will define a tool list that the fine-tuned model can call. Each tool must have:
- A stable tool name (must appear in the schema).
- A JSON arguments schema (types and required keys).
- A deterministic Unity-side implementation.

Minimum tool set for "iteration speed":
1. File tools
   - `read_file` (Unity can implement server-side file reads if backend has project path, otherwise request file contents explicitly)
   - `apply_patch`
   - `write_file`
   - `create_file`
   - `append_to_file`
2. Scene/GameObject tools
   - `get_scene_hierarchy` (return hierarchy snapshot so the model can reference paths)
   - `create_game_object`
   - `add_component`
   - `remove_component`
   - `set_component_property` (typed set, map to serialized fields / properties)
3. Event wiring tools
   - `connect_ui_event` (Button.onClick or UnityEvent wiring)
4. Compile/test tools
   - `collect_compile_errors` (capture CS compiler output from Unity or batch compiler)
   - `run_unity_editor_tests` (optional)

### 5.2 Tool execution rule
- Backend returns tool calls, but the Unity plugin is responsible for executing them.
- Backend should be treated as "planner/executor-orchestrator", not as authority over editor state.

## 6. Prompting + Tool Loop

### 6.1 RAG agent loop (server tools loop)
Use `/query_stream_with_tools` with tool loop enabled:
1. Backend streams answer text.
2. Backend appends `__TOOL_CALLS__` followed by a JSON array of tool calls.
3. Backend appends `__USAGE__` (context usage stats).
4. Unity plugin executes tool calls in order and records them in Pending & Timeline.

### 6.2 Composer fine-tuned loop (single-turn)
Use `/composer/query` or streaming variants:
- Fine-tuned model emits tool calls directly in one response.
- Unity parses XML blocks to tool-call JSON and executes them.

## 7. Data Building with OpenAI Calls

### 7.1 Dataset format
The Composer v3 pipeline expects records shaped like:
- JSONL input files containing `{ "messages": [system, user, assistant] }`.
- Assistant content must be strict:
  - AGENT mode: includes one or more `<tool_call>...</tool_call>` blocks.
  - ASK mode: exactly one question mark terminated question, and no tool-call blocks.

### 7.2 Generator approach (OpenAI calls)
Implement scripts that:
1. Load `tools.json` (repo root) (or generate it first from backend tool registry).
2. Use the strict system prompt formatting rules from Composer v3.
3. Request the model to emit:
   - AGENT outputs: tool calls required
   - ASK outputs: single question only
4. Validate outputs using the Composer v3 dataset validators (or normalize by rebuilding strict datasets and discarding rejects).

### 7.3 Strict normalization and rejection
Use existing scripts (Composer v3) to:
- rebuild strict AGENT and ASK datasets
- run validation + audit + release gating

### 9.1 Pre-train gates
Run:

### 9.2 Post-train release gate
Run:
- `python fine_tuning/scripts/v3/run_composer_v3_posttrain_release_gate.py`

### 9.3 Inference contract suite
Run:
- `python -m fine_tuning.testing.composer_v3_inference_contract_suite`

Acceptance for "release success":
- unknown tool calls count is zero
- malformed tool blocks count is zero
- AGENT produces tool calls
- ASK produces exactly one question and no tool calls

## 10. Implementation Plan (Milestones)

### Milestone A: Bootstrap Unity editor plugin skeleton (1-2 weeks)
1. Create Unity plugin package:

   - `fine_tuning/scripts/v3/run_composer_v3_pretrain_gates.py`
